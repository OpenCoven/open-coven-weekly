<div style="margin-top:1rem;max-width:960px">

**Largest revision of the Model Context Protocol since launch.** Delivers on the 2026 roadmap:

- 🌐 **Stateless core** — scales on ordinary HTTP infra (no sticky sessions)

- ⏱️ **Tasks extension** — long-running work, properly modeled

- 📜 **Formal deprecation policy** — the protocol can evolve without breaking what you've built

**Plus:** Enterprise-Managed Authorization (EMA) extension is now stable. Centrally manage auth across all MCP servers. **Adopted by Anthropic, Microsoft, Okta.**

</div>

<!--

<div class="label">Security</div>

# Two ways the agent<br/>stack got *owned*.

<div style="margin-top:1rem;max-width:960px">

Two security stories this week. Read both.

- 💥 **AutoJack RCE** in AutoGen Studio — localhost WebSocket trust + missing MCP API auth + unsanitized base64 params = arbitrary shell from a malicious webpage. Fixed in v0.4.8 with gVisor sandboxing + Seccomp. **Microsoft moved AutoGen to community maintenance; new deployments should use Microsoft Agent Framework 1.0.**
- 🕵️ **Fake MCP skill bypassed security scans** — AIR built a malicious MCP skill, passed standard vetting, mutated the payload *after approval*. Propagated to **~26,000 agents** before detection.

**The pattern:** "scan once, trust forever" doesn't work for agent tooling. Payload mutability after vetting is a structural problem.

</div>

<!--
Two security stories that anyone running agents needs to know about.

First — AutoJack. A vulnerability chain in AutoGen Studio that let a malicious webpage trigger remote code execution. The chain was localhost WebSocket trust, missing auth on MCP API routes, and unsanitized base64 params — combined, the agent would execute arbitrary shell commands. Fixed in v0.4.8 with gVisor sandboxing and Seccomp profiles. The big related news: Microsoft has officially moved AutoGen to community maintenance mode. New enterprise deployments should be on Microsoft Agent Framework 1.0.

Second — the fake MCP skill story. Security firm AIR published a write-up on June 23. They built a fake MCP-compatible skill, passed it through standard security vetting, then *altered the payload after approval*. The malicious version propagated to roughly twenty-six thousand agents, including corporate accounts, before detection.

These are two flavors of the same pattern. The agent tooling ecosystem still treats security like web app stores treated it in 2008 — scan on submission, trust forever. That model is broken for agents because the payload is the program is the prompt is the executable, and any of those can mutate after vetting.

If you're running MCP servers in production, treat them like supply chain. Pin versions. Re-verify on update. Sandbox aggressively.
-->

<div class="label">Identity</div>

# Linux Foundation:<br/>*name service* for agents.

<div style="margin-top:1rem;max-width:960px">

The Linux Foundation announced intent to launch the **Agent Name Service (ANS)** — a DNS-anchored identity layer for AI agents.

- 🌍 Built on existing DNS infrastructure (millions of queries/sec, already global)
- 🪪 Verifies: who an agent represents
- 🔑 Verifies: what permissions it has
- 📜 Verifies: code + operational history are unchanged

**Why now:** as agents proliferate across enterprise networks, identity + provenance becomes table stakes. This is the foundation trying to standardize *before* it becomes a mess of proprietary solutions.

</div>

<!--
The Linux Foundation announced intent to launch the Agent Name Service this week. ANS, for short. It's a DNS-anchored identity layer for AI agents.

The pitch is that agents are about to proliferate inside enterprise networks at a scale that needs real identity and provenance verification. Who does this agent represent. What permissions does it have. Is its code and operational history what we last verified.

And they're building it on DNS — which is interesting, because DNS already handles millions of queries per second at planetary scale, and the trust hierarchy is already well-understood by every enterprise IT team on earth.

This is the foundation trying to get out ahead of the mess — the alternative is every vendor shipping their own proprietary identity scheme and us all paying the integration tax for ten years. Worth tracking.
-->
